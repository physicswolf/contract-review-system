# 模块一：上传与结构解析

## 1. 概况

本模块负责合同文件的安全接收、校验落盘、异步解析以及将线性文档还原为结构化章节树。用户上传一份 `.docx` 或 `.pdf` 合同后，系统经过多层安全校验将文件保存到本地，随后在后台线程中调用解析引擎生成 `document.json`，其中 `contract_structure` 字段包含完整的合同条款层级树。

**核心能力**：
- 文件安全校验（文件名、MIME、文件头/容器签名、大小限制、双扩展名防伪）
- 异步解析（上传即返回任务句柄，后台线程执行，前端轮询状态）
- 多引擎支持（DOCX：docling / python-docx；PDF：docling / pymupdf4llm）
- 合同编号识别（支持 part/chapter/article/dotted/arabic/chinese/parenthesized 七种编号体系）
- 层级树构建（基于 rank 权重的栈式组树算法）

## 3. 设计

### 3.1 安全校验链

文件从接收到落盘经过五道校验，任一步失败即拒绝并清理临时文件：

```
原始文件名 → 控制字符/路径穿越/双扩展名 → MIME白名单 → 分块写入+大小限制 → 文件头签名(PDF:%PDF- / DOCX:zip+Content_Types+word/)
```

- 上传写入 `.part` 临时文件，全部校验通过后 `rename` 转正
- 大小限制实行**双重防护**：中间件读 `Content-Length` 粗筛（+1024字节容差），落盘期逐块累加精确判定

### 3.2 解析引擎路由

```
parse_uploaded_document(source, file_id, ext, settings)
  │
  ├─ .docx + python-docx → 直接读DOCX段落/编号 → 章节树 → document.json
  │
  ├─ .docx + docling → LibreOffice转PDF → Docling解析 → 章节树 → document.json (+ document.pdf)
  │
  ├─ .pdf + docling → Docling解析 → 补充章节树 → document.json
  │
  └─ .pdf + pymupdf4llm → PDF→Markdown分页块 → 章节树 → document.json
```

### 3.3 章节树构建算法

核心流程：所有引擎先将文档转为统一的 `StandardLine` 列表，然后交由共用的 `parse_contract_structure_from_lines` 构建树。

```
文本行列表
  │
  ├─ 清洗 → 过滤（封面/目录/页码/表格标记行）
  │
  ├─ 逐行编号识别（parse_number_token）
  │   匹配优先级：第X部分 > 第X章 > 附件X > 第X条 > 一、> 1.1 > 1、>（一）
  │
  ├─ 栈式组树（attach_node）
  │   新节点rank ≤ 栈顶rank → 向上弹栈 → 找到rank更小的父节点 → 挂载
  │
  └─ 正文挂载（append_content）
      非标题行 → 追加到栈顶节点的 content[]
```

**编号类型与层级权重**：

| kind | 示例 | rank | 层级 |
|------|------|------|------|
| part | 第一部分 | 0 | 最高 |
| chapter | 第一章 | 0 | 最高 |
| attachment | 附件一 | 5 | 高 |
| article | 第一条 | 10 | 中 |
| chinese | 一、 | 10 | 中 |
| arabic | 1、 | 20 | 低 |
| dotted | 1.1 / 1.1.1 / 1.1.1.1 | 21~23 | 更低 |
| parenthesized | （一）/（1） | 40 | 最低 |

### 3.4 任务状态机

```
queued ──▶ running ──▶ succeeded
   │           │
   └───────────┴──▶ failed
```

- `succeeded` 时记录 `parsing_dir` 和 `document_json_path`
- `failed` 时清理上传文件 + 半成品解析目录，记录 error.code/message
- 状态存于进程内存，服务重启丢失（Demo 取舍）

## 4. 接口

### 4.1 上传文件

```
POST /api/v1/files/upload
Content-Type: multipart/form-data
```

**输入**：`file`（form field，必填，.docx 或 .pdf）

**响应 200**：
```json
{
  "success": true,
  "data": {
    "file": {
      "id": "uuid",
      "original_name": "合同.pdf",
      "extension": ".pdf",
      "content_type": "application/pdf",
      "size": 123456,
      "uploaded_at": "2026-06-25T10:30:00+08:00"
    },
    "task": {
      "id": "task-uuid",
      "file_id": "uuid",
      "status": "queued",
      "stage": "queued",
      "created_at": "2026-06-25T10:30:00+08:00",
      "updated_at": "2026-06-25T10:30:00+08:00",
      "error": null,
      "parsing_dir": null,
      "document_json_path": null
    }
  }
}
```

**错误码**：

| HTTP | code | 场景 |
|------|------|------|
| 413 | FILE_TOO_LARGE | 超过 50MB |
| 415 | UNSUPPORTED_FILE_TYPE | 扩展名或 MIME 不支持 |
| 415 | FILE_SIGNATURE_MISMATCH | 内容与扩展名不匹配 |
| 400 | EMPTY_FILE | 空文件 |
| 400 | INVALID_FILENAME | 路径穿越/控制字符/双扩展名 |
| 422 | FILE_REQUIRED | 未提交文件 |

## 5. 数据

### 5.1 上传文件存储

```
storage/uploads/YYYY/MM/DD/
├── {file_id}.pdf          ← 原始文件（UUID命名，非用户原始名）
└── {file_id}.json         ← FileMeta 元数据
```

- 按上海时区日期分目录
- 文件名使用 UUID，防御路径遍历
- 写入时使用 `.part` 临时后缀，完成后原子 rename

### 5.2 解析产物

```
storage/parsing/{file_id}/
├── document.json           ← 解析产物（必有）
└── document.pdf            ← DOCX→Docling路径的转换PDF（可选）
```

  "schema_name": "DoclingDocument | PyMuPDF4LLMContractDocument | PythonDocxContractDocument",
  "name": "file_id",

  "tables": [ ... ],

    "node_id": "root",

      {
        "node_id": "node_0001",
