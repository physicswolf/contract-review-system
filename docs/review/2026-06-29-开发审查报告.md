## 三、审查发现的问题

### 🔴 严重（1 个）

**修复**：从原始后端目录 `/home/wolfsoft/my_code/contract/homepage/backend/tests/` 恢复测试文件，并检查与新 API 的兼容性。

### 🟡 中等（3 个）

**2. `/api/dimensions` 列表和详情返回格式不一致**

当前 `dimensions.py` 中只有列表（`@router.get("")`）使用了 `_tmp_ok()` 格式 `{code:0, data:[...]}`，但创建 (`POST`) 和更新 (`PUT`) 也使用了 `_tmp_ok()`（正确）。而 `/names` 和 `/selectable` 仍返回扁平格式。**这会导致**：前端 `dimension.service.js` 的 `adapt()` 函数在调用 list/create/update 时能正确解析 `data.data`，但 selectable 走的是直接返回，不经过 adapter。需要确认前端代码是否兼容。

**3. `/api/contract-types` 错误响应格式不统一**

`contract_types.py` 中错误返回使用了 `error_response(500, "INTERNAL_ERROR", "合同类型列表读取失败")`，返回 `{"detail":"..."}` 格式。但前端 `type.service.js` 期望 `{code:xxx, msg:"..."}`。tmp 的 `http.js` 能同时读 `data.msg || data.detail`，所以错误能显示，但 HTTP 状态码和 code 值不一致。

**4. `/api/audit_points.py` 的 `delete_by_id` 对已删除记录行为**

当前代码先检查 `record is None` 再调 `delete_by_id`，但 `delete_by_id` 只查 `deleted_at IS NULL` 的记录。如果记录已被软删除，第二次调用会返回 `False`，后端返回 404。这是正确的行为，与 tmp Flask 版本一致。通过。

### 🟢 轻微（2 个）

**5. `audit.py` 仍存在但不注册路由**

`api/audit.py` 文件仍存在于目录中但 `main.py` 不再注册。是一个无用文件。

**6. 响应格式混用**

项目中同时存在三种响应格式：
- `{code:0, msg:"ok", data}` — tmp 兼容端点（`audit_points.py`, `dimensions.py` 列表, `contract_types.py`）
- `{items, total}` / `{ok:true}` / 直接对象 — 旧端点（`points.py`, `contracts.py`, `auth.py`）
- `{"detail":"..."}` — 错误响应（所有错误走 `error_response()`）

建议在文档中明确标注哪些端点使用哪种格式。
