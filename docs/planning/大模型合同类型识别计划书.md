## 一、背景

### 1.1 现状（来自 Codex 巡检结论）

| 功能 | 当前实现 | 问题 |
|------|---------|------|
| 上传后合同类型识别 | `contract_extractor.py` 的**关键词规则 + 文件名** | 识别不准，大量返回"未分类" |
| 审核结果生成 | DB 审查点 `risk_points` + `default_result` 降级 | 无 AI 研判 |
| LLM 配置 | `config.py` 中 `LLM_API_URL/KEY/MODEL_NAME` 占位 | 未使用 |

### 1.2 本轮目标

仅改造**上传后合同类型识别**场景。审核结果 LLM 研判不在本轮范围。

## 二、方案概要

### 2.1 数据来源

| 输入 | 来源 | 获取方式 |
|------|------|---------|
| 合同首部正文 | 文档解析后的 `document.json` → `contract_structure` 树 → `preamble` 节点的 `content[]` 或根节点直属正文 | `services/structure_editor.py` 的 `load_document_json()` |
| 已启用合同类型清单 | `contract_type` 表 `WHERE enabled=1 AND deleted_at IS NULL` | `contract_type_repository.find_all({"enabled": 1})` |

### 2.2 Prompt 构造

```
系统指令: 你是合同类型识别助手。根据合同首部内容判断合同类型，只能从给定清单中选择一项。
可选清单: [采购合同, 服务合同, 销售合同, 租赁合同, 合作协议, 补充协议]
合同首部: [甲方：XX公司\n乙方：YY公司\n...]
输出要求: 只返回类型名称，不返回其他内容。
```

### 2.3 LLM 调用

```
POST {LLM_API_URL}
Header: Authorization: Bearer {LLM_API_KEY}
Body: { model: LLM_MODEL_NAME, messages: [{role:"system", content: system_prompt}, {role:"user", content: user_prompt}], temperature: 0 }
```

解析返回的 `choices[0].message.content`，校验是否为清单中的类型。

### 2.4 与上传流程的衔接点

当前上传响应 `POST /api/contracts/upload` 返回：`{ id, name, detectedType, matchConfidence, fileId, enableStructureEditor }`

改造后：`detectedType` 从 LLM 推断获取（替代原有的关键词规则）。若 LLM 调用失败，降级为原有的关键词规则。

## 四、降级策略

| 场景 | 处理 |
|------|------|
| LLM_API_URL 未配置 | 跳过 LLM，使用原有关键词规则 |
| LLM 返回超时（>10s） | 降级为关键词规则 |
| LLM 返回的类型不在清单中 | 降级为关键词规则 |
| LLM 返回 HTTP 错误 | 降级为关键词规则 |
| 合同首部提取失败（无 preamble 节点） | 使用 document 根节点的 content[] 作为正文 |
