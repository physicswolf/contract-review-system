## 一、背景与目标

当前系统已采用 **LiteParse 默认 + 规则切分** 完成 PDF/Word 解析与章节结构化，但实测暴露两个痛点：

1. **Word 路径不稳定**：LiteParse 依赖 LibreOffice 把 docx 转 PDF，再解析。本次复测中，46KB 的 docx 直接 **转换超时（>120s 失败）**，生产不可接受；
2. **PDF 章节切分过度**：基于字号的标题识别不鲁棒，召回前的章节粒度噪声大。

因此需要重新横向对比 6 个主流框架，目标是回答三个问题：

- **谁负责 Word**：找一条快、稳、可拿到原生样式（标题层级、表格）的 docx 路径；
- **谁负责 PDF**：在「速度」与「结构化（Markdown/标题/表格）」之间取舍；
- **是否需要重型模型**：MinerU / docling 这类带版面模型的方案，对合同场景是否值得引入。

| 项       | 配置                               |

| 计时方法    | 每文件解析取最优值（剔除首次冷启动/模型加载）          |
| Word 依赖 | LibreOffice（仅 LiteParse 需要）      |


测试数据取自项目 `backend/uploads/` 的真实合同样本：


| 编号    | 类型   | 体量           | 说明                               |
| ----- | ---- | ------------ | -------------------------------- |
| PDF-A | PDF  | 922KB / 24 页 | 电子版合同                            |
| PDF-B | PDF  | 1.6MB / 35 页 | 电子版合同（含较多表格）                     |
| DOCX  | Word | 46KB / 200 段 | 电子版合同（**未使用 Word 标题样式**，含 3 个表格） |


> 说明：python-docx / PyMuPDF / pymupdf4llm / LiteParse / **docling** 均为**本机实跑**（Apple Silicon Mac，纯 CPU，无 GPU）取得真实数据；MinerU 模型权重 GB 级，本次未在本机安装，以官方能力 + 公开基准给出结论（其同样支持 CPU 本地运行，仅更重更慢）。

## 三、实测数据（本机真实结果）

### 3.1 PDF 解析


| 框架              | PDF-A（24页） | PDF-B（35页） | 输出形态           | 结构化信息                                   |
| --------------- | ---------- | ---------- | -------------- | --------------------------------------- |
| **PyMuPDF**     | **142 ms** | **218 ms** | 纯文本            | 无（仅文本，需自行重建版面）                          |
| **LiteParse**   | **47 ms**  | **89 ms**  | 纯文本 + 字号/坐标元数据 | 字号/bbox（需规则二次加工）                        |
| **pymupdf4llm** | 1540 ms    | 2808 ms    | Markdown       | PDF-A: 36 标题/13 表格行；PDF-B: 70 标题/75 表格行 |


文本量（字符数，越接近真实正文越完整）：


| 框架          | PDF-A  | PDF-B  |
| ----------- | ------ | ------ |
| PyMuPDF     | 12,644 | 24,339 |
| LiteParse   | 13,880 | 28,521 |
| pymupdf4llm | 13,469 | 27,771 |


**结论**：

- **速度**：LiteParse ≈ PyMuPDF（毫秒级，纯文本路径），pymupdf4llm 慢 1~2 个数量级（默认开启版面模型 + 按需 OCR，onnxruntime 推理）；
- **结构化**：pymupdf4llm 直接产出带 `#` 标题层级与 Markdown 表格的输出，是三者中**唯一开箱给到 LLM 友好结构**的；LiteParse/PyMuPDF 都需要自己写规则重建标题与表格；
- **文本完整度**：三者相近，pymupdf4llm/LiteParse 因保留表格/换行结构，字符数略高于 PyMuPDF 的裸文本。

### 3.2 Word 解析


| 框架              | DOCX（46KB/200段） | 结果                             |
| --------------- | --------------- | ------------------------------ |
| **python-docx** | **22 ms**       | 成功：200 段 / 3 表格 / 标题样式 0 个     |
| **LiteParse**   | —               | **失败（LibreOffice 转换超时 >120s）** |


**结论**：

- **python-docx 是 Word 的正确路径**：毫秒级、零外部依赖、可直接拿到段落、**表格**、**原生标题样式**；
- **LiteParse 的 Word 路径不可用于生产**：本次直接超时失败，且即便成功也要走 LibreOffice，慢且重；
- 一个关键发现：**该合同样本 `Heading 样式 = 0`** —— 很多实际合同并不用 Word 内置标题样式，而是用「加粗 / 字号 / 居中 / 手工编号」。这意味着 Word 章节切分**不能只依赖样式**，需叠加「编号正则 + 加粗/字号」启发式（与现有 splitter 思路一致）。

### 3.3 章节切分效果对比（本项目核心关注）★

把**同一份合同 PDF-A** 喂给各框架，抽取各自识别出的「章节标题清单」做定性对比。该合同真实结构为：**抬头（甲方/乙方/鉴于）+ 第一条～第十七条**，期望切出约 18~20 个章节。

| 框架 | 识别标题数 | 效果 | 具体表现 |
|------|------|------|------|
| **docling** | 24 | ✅ **最佳** | 第一～十七条**全部正确识别**，抬头也对，仅 2 处噪声；层级合理 |
| **pymupdf4llm** | 36 | 🔶 中等 | 条款全召回，但混入「甲方：/地址：/邮政编码：/失。/害、篡改；」等正文碎片（约一半为噪声标题） |
| **LiteParse + 现有规则** | 131 | ⚠️ 过度切分 | 「第X条」识别对，但把 `1.1 / 4.1.1` 等**子条款全部当成标题**，粒度过碎，对召回是噪声 |
| **PyMuPDF + 字号规则** | 2 | ❌ 失效 | 正文与条款标题**同字号（均 14pt）**，纯字号规则识别不出「第X条」 |
| **python-docx（docx样本）** | 28 | 🔶 依赖样式 | 该样本 Heading 样式=0，纯靠正则→只切出子条款，**未识别出「第X条」主标题** |

**docling 识别结果（节选，干净准确）**：
```
第一条 定义 / 第二条 采购模式 / 第三条 采购计划 / … / 第十七条 附件 / （以下无正文）
```

**LiteParse 现有规则结果（节选，过度切分）**：
```
第二条 采购模式
  2.1 甲方向乙方下达加盖公章的《采购订单》…   ← 子条款被当成 L3 标题
  2.2 乙方确认接收《采购订单》…
  2.3 甲方未授权任何人员…
```

**章节切分关键结论**：

1. **章节切分效果 docling 明显最优** —— 它用**模型识别标题**（`section_header`），既不靠字号也不靠正则，因此在「标题与正文同字号」的真实合同里依然能干净切出第一～十七条。这是六者中唯一开箱即得高质量章节结构的方案。
2. **字号方案在合同场景基本不可靠** —— 本合同条款标题与正文字号完全相同，PyMuPDF 纯字号只切出 2 个，LiteParse 的字号辅助也失效。
3. **LiteParse 现有规则过度切分** —— 把 `4.1.1` 级子条款都当标题（131 个），章节太碎，需限制切分层级（只在「第X条」级别断章）。
4. **docx 章节切分取决于是否使用 Word 标题样式** —— 规范使用 Heading 样式时 python-docx 最准；本样本未用样式，效果打折，必须叠加编号正则。
5. **pymupdf4llm 居中可用** —— 条款能全召回，但需加一道「噪声过滤」（剔除「甲方：」「地址：」类伪标题、过短/以标点结尾的行）才能用于生产。

> 章节切分选型建议：本项目**对准确率要求高，PDF 默认采用 docling（模型识别标题，切分质量最佳）**；**高吞吐/轻量备选 → pymupdf4llm + 噪声过滤**；**Word 规范文档 → python-docx 样式**；**纯字号/纯正则方案（PyMuPDF/LiteParse 现状）在合同场景不可靠，不作主路径**。

## 四、六框架逐项评估

### 1. python-docx —— Word 的「原生结构读取器」

- **定位**：纯 Python 读写 `.docx`，直接访问段落、Run（加粗/字号/字体）、表格、样式、章节。
- **优势**：极快、零依赖、可拿到**最高保真的 Word 结构**（标题样式、表格单元格、编号）；写回也支持。
- **短板**：**仅支持 docx**（不支持 doc、不支持 PDF）；不做版面/阅读顺序推断（docx 本身有序，无需）。
- **合同场景**：✅ **Word 路径首选**。表格、标题样式、编号都能拿到，配合规则切分效果最好。

### 2. PyMuPDF（fitz）—— PDF 的「高速底层引擎」

- **定位**：MuPDF C 引擎的 Python 绑定，PDF 文本/图片/坐标提取 + 编辑（合并/拆分/标注/脱敏）。
- **优势**：**最快**（毫秒级）、可拿字体/坐标/颜色等细粒度信息、功能全（不止解析）。
- **短板**：输出是**裸文本**，表格/标题/阅读顺序需自己实现；扫描件需外接 OCR；**许可证 AGPL-3.0**（商用需注意，内部研究无碍）。
- **合同场景**：✅ 适合做**高速兜底**与**底层能力**（取坐标/字号供切分、PDF 操作），但单用它结构化成本高。

### 3. pymupdf4llm —— PyMuPDF 之上的「Markdown 结构化层」

- **定位**：PyMuPDF 官方扩展，一行调用把 PDF 转 **Markdown/JSON**，面向 RAG/LLM 摄取；带版面分析、表格转 Markdown、字号映射标题、按页分块、按需 OCR。
- **优势**：**开箱即得 LLM 友好结构**（标题层级 + Markdown 表格 + 阅读顺序），无需 GPU、无需联网；与 LlamaIndex/LangChain 原生集成。
- **短板**：比裸 PyMuPDF 慢 1~~2 个数量级（实测 1.5~~2.8s/文件）；复杂/无边框表格、合并单元格仍会丢失；继承 AGPL。
- **合同场景**：🔶 **PDF 轻量/高吞吐备选**。秒级延迟、零部署负担，适合吞吐压力大或不便部署模型的场景；但章节标题有噪声/偶有阅读顺序错乱，准确率优先时让位 docling。

### 4. LiteParse —— 当前在用的「Rust 本地解析库」

- **定位**：LlamaIndex 系开源本地解析（Rust 内核 + Python 绑定），PDF 直读快，Word 经 LibreOffice 转 PDF。
- **优势**：PDF 解析**极快**（毫秒级），输出带字号/坐标元数据；本地、可离线。
- **短板**：**Word 路径依赖 LibreOffice，本次实测超时失败**，不稳定且重；结构化仍需规则二次加工；生态/文档相对小众。
- **合同场景**：⚠️ PDF 可继续作为高速兜底，但 **Word 路径建议下线**，改用 python-docx。

### 5. docling（IBM，本机实测）—— 「带版面模型的结构化解析」

- **定位**：IBM 开源（MIT），PDF/DOCX/PPTX/XLSX/HTML → 统一 `DoclingDocument`，导出 Markdown/JSON/HTML，含版面、阅读顺序、表格结构、公式、OCR；可选 Granite-Docling 258M 视觉模型。
- **本机实测（Apple Silicon，纯 CPU，版面模型，OCR 关，模型已缓存）**：
  - PDF-A 24 页 → **8.9s（0.37s/页）**，识别 24 个标题，13,303 字符；
  - PDF-B 35 页 → **14.1s（0.40s/页）**，识别 67 个标题，29,281 字符。
- **优势**：**结构保真度高**（多栏、表格、阅读顺序均有学习模型支撑），标题为模型识别而非字号启发；多格式统一、Markdown 优先；**纯 CPU 可跑，整篇合同十几秒出结果**。公开基准表格 TEDS≈0.887、阅读顺序≈0.90。
- **短板**：**安装重**（含 torch 等依赖），**首次需联网下载模型**（数百 MB，受限网络下可能很慢——本次首跑因下载耗时 30 分钟+，缓存后仅 9s）；开启表格结构模型（TableFormer）需额外下一个模型且推理更慢；扫描件需 OCR 会进一步变慢（CPU 下建议上 GPU）。
- **合同场景**：✅ **PDF 默认解析器（准确率优先）**。章节切分质量最佳、阅读顺序正确，纯 CPU 即可（模型预缓存后内网离线运行）；复杂表格开 TableFormer、扫描件开 OCR 时建议上 GPU。

### 6. MinerU（资料分析）—— 「高精度版面/OCR 解析」

- **定位**：高精度文档解析引擎，强于复杂版面、扫描件 OCR、表格、公式，输出结构化 Markdown/JSON。
- **优势**：**复杂版面与扫描件解析精度最高**梯队；适合「图文混排、扫描合同、含公式/复杂表格」的硬骨头。
- **能否本地跑**：✅ **可以**。官方提供 CPU 模式，纯 CPU 也能本地运行（与 docling 同理，首次需下载 GB 级模型权重）；**GPU 是为速度而非可行性**。本次未在本机安装（模型体量大 + 受限网络下载耗时过长）。
- **短板**：**六者中最重**（模型 GB 级，CPU 下延迟最大）、部署/资源成本高；对纯电子版文字合同属「杀鸡用牛刀」。
- **合同场景**：🔶 **专项兜底**。仅当出现扫描件 / 复杂版面失败样本时定向路由；CPU 可应急，批量/低延迟需求上 GPU。

## 六、选型方案

### 6.1 推荐架构：分格式 + 分难度的「双层路由」

> **决策：准确率优先，PDF 默认使用 docling。** 合同审核对章节切分/结构准确率要求高，而 docling 在章节切分实测中质量最佳（模型识别标题、阅读顺序正确、几乎无噪声）；其纯 CPU 即可运行（0.37s/页），延迟可接受。pymupdf4llm 降为「高吞吐/轻量」备选。

```
                       ┌─────────────────────────────┐
          .docx  ──────▶  python-docx（原生结构）       │  ← Word 首选（快/稳/拿样式表格）
                       └─────────────────────────────┘
                                     │ 扫描/无样式 PDF 另存
                                     ▼
   .pdf  ─────────────▶  docling（模型识别标题，准确率优先）  ← PDF 默认
                                     │
                       ┌────────────┴────────────┐
            高吞吐/轻量需求                  扫描件/复杂版面/公式
                  │                                  │
                  ▼                                  ▼
       pymupdf4llm / PyMuPDF（轻量备选）        MinerU（重型兜底，建议 GPU）
```

### 6.2 三句话结论

1. **Word 用 python-docx**：替换当前 LiteParse 的 Word 路径，解决超时与不稳定，并直接拿到标题样式 + 表格供切分；
2. **PDF 默认 docling（准确率优先）**：用其模型识别的标题做章节切分，质量最佳、阅读顺序正确；纯 CPU 可跑（0.37s/页，整篇合同十几秒），模型预缓存后可离线；
3. **轻量/重型按需**：高吞吐或极轻部署场景用 pymupdf4llm / PyMuPDF；扫描件/复杂版面用 MinerU 定向兜底（建议 GPU），不进默认链路。

### 6.3 与现有系统的改造点


| 模块                     | 现状                      | 调整建议                                                                                     |
| ---------------------- | ----------------------- | ---------------------------------------------------------------------------------------- |
| `services/parser.py`   | LiteParse 统一入口          | 改为按扩展名分发：docx→python-docx，pdf→docling（保留 pymupdf4llm/PyMuPDF/LiteParse 兜底开关）              |
| `services/splitter.py` | 字号 + 样式 + 正则            | PDF 改用 docling 的模型标题（section_header）作为主信号；Word 用 python-docx 样式 + 编号正则（样本样式可能为 0，必须叠加正则） |
| `requirements.txt`     | liteparse + python-docx | 增加 `docling`（含模型，需预缓存权重）；`pymupdf`/`pymupdf4llm` 作轻量备选；LiteParse 降级为可选兜底                |
| 模型权重                   | 无                       | docling 版面/表格模型一次性离线拷贝进主机（内网出网受限）；MinerU 以独立服务（GPU）暴露 HTTP 做扫描件兜底                        |


### 6.4 落地路线

- **阶段一（即可做）**：Word 切到 python-docx；PDF 切到 docling（模型标题做章节切分），预缓存 docling 模型权重。纯本地、无 GPU。
- **阶段二（按需）**：补一个「解析质量校验」（空文本/表格丢失/疑似扫描件 → 触发兜底），高吞吐场景用 pymupdf4llm 加速、复杂表格开 docling TableFormer。
- **阶段三（专项）**：出现扫描件/复杂版面失败集后，在 GPU 主机部署 MinerU 定向兜底。

---

## 七、风险与注意事项

1. **许可证**：PyMuPDF / pymupdf4llm 为 **AGPL-3.0**。内部研究/自用无碍；若未来对外分发 SaaS，需评估合规（或改用 MIT 系的 docling）。
2. **pymupdf4llm 延迟**：默认开启版面模型，1.5~2.8s/文件。若批量摄取量大，可关闭 layout 模式或仅对疑难页启用。
3. **复杂表格仍会丢**：pymupdf4llm 对合并单元格/无边框表格不完美，强表格需求才上 docling/MinerU。
4. **docx 标题样式不可依赖**：实测样本 Heading=0，章节切分必须叠加编号正则与加粗/字号启发式。
5. **docling 作默认的代价**：docling 首次需联网下载版面/表格模型（数百 MB，内网部署须**预先缓存模型权重**）。纯 CPU 实测 0.37s/页、整篇合同十几秒，对「上传即解析」可接受；若吞吐压力大或延迟敏感，可对部分文档走 pymupdf4llm 加速，或上 GPU。MinerU 体积更大（GB 级），仅作扫描件/复杂版面兜底。
