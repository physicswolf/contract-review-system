[# 审核详情页技术设计文档

## 1. 概要

### 1.1 页面功能描述

审核详情页分为左右两栏：

- **左栏**：用 **pdf.js** 渲染合同原文 PDF，本期不实现文档批注功能
- **右栏**：审查结果面板，按「审核维度（Tab）× 风险级别（过滤）」两个轴组织结果

每张结果卡片包含：风险标题、风险等级、风险概述、修改方案、所属条款及维度标签。

### 1.2 核心设计决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 左侧原文渲染 | pdf.js 直接渲染 PDF | 后端提供文件 URL，无需解析文本结构 |
| 文档批注 | 本期不实现 | - |
| 文档解析工具 | docling | 统一将各格式转为结构化 Markdown/文本供 LLM 使用 |
| 格式转换链路 | docx/doc → PDF → docling 解析 | .doc 先用 LibreOffice 转 docx，再转 PDF；PDF 直接 docling 解析 |
| LLM 文本存储 | 按行拆分存 JSON，仅供 Prompt 构造 | 前端不使用 |
| 大模型调用 | 按维度（rule_category）分批调用 | 每个维度独立 Prompt，结果逐批写入 |

### 1.3 存储分工

| 存储 | 存放内容 |
|------|----------|
| 文件存储 | 原始上传文件、转换后 PDF、docling 解析的行数组 JSON（供 LLM） |
| MySQL | 合同元信息、审核任务、审核结果条目、维度统计 |

## 3. 文件格式转换链路

```
用户上传
  ├── .pdf  ──────────────────────────────────→ docling 解析 → lines.json
  ├── .docx ──→ LibreOffice 转 PDF ──────────→ docling 解析 → lines.json
  └── .doc  ──→ LibreOffice 转 docx → 转 PDF → docling 解析 → lines.json

转换后 PDF 统一用于：
  1. pdf.js 前端渲染（左侧原文展示）
  2. 存档备用
```

## 4. 存储设计

### 4.1 文件存储路径规范

以 `task_id` 为目录单位，每个任务文件独立存放：

```
contracts/
└── task_{task_id}/
    ├── original.{ext}      # 原始上传文件（保留原格式）
    ├── converted.pdf       # 统一转换后的 PDF（前端 pdf.js 渲染用）
    └── blocks.json         # docling 解析的文本块（含 PDF 坐标，供 LLM + 前端定位）
```

### 4.2 blocks.json 格式

docling 解析 PDF 后，每个文本块输出 PDF user space 坐标（单位：points，原点左下角）：

```json
{
  "total_pages": 18,
  "blocks": [
    {
      "no": 1,
      "page": 1,
      "text": "智慧园区软件采购合同",
      "bbox": { "x": 200, "y": 750, "width": 180, "height": 20 }
    },
    {
      "no": 3,
      "page": 1,
      "text": "甲方：华城数字建设有限公司",
      "bbox": { "x": 90, "y": 710, "width": 150, "height": 16 }
    },
    {
      "no": 10,
      "page": 4,
      "text": "若甲方在十个工作日内未提出书面异议，则视为项目自动验收通过。",
      "bbox": { "x": 90, "y": 600, "width": 400, "height": 16 }
    }
  ]
}
```

- `bbox` 坐标为 PDF user space（points），原点左下角，与 pdf.js viewport 坐标系一致
- 同时用于：① 构造大模型 Prompt（传 `no` + `text`）；② 前端高亮定位（传 `page` + `bbox`）

### 4.3 MySQL 表设计

#### 合同文件表 `contract_file`

```sql
CREATE TABLE contract_file (
  id              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  name            VARCHAR(500)    NOT NULL COMMENT '合同名称',
  party_a         VARCHAR(200)    COMMENT '甲方名称',
  party_b         VARCHAR(200)    COMMENT '乙方名称',
  contract_type   VARCHAR(50)     COMMENT '合同类型',
  original_ext    VARCHAR(10)     NOT NULL COMMENT '原始上传格式 doc/docx/pdf',
  created_at      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  deleted_at      DATETIME        DEFAULT NULL,
  PRIMARY KEY (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> 文件路径通过 `task_id` 推导，不再单独存 file_key 字段。

#### 审核任务表 `audit_task`

```sql
CREATE TABLE audit_task (
  id                 BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  contract_file_id   BIGINT UNSIGNED NOT NULL,
  -- 文件路径通过 id 推导：contracts/task_{id}/
  parse_status       TINYINT  NOT NULL DEFAULT 0 COMMENT '文件解析状态 0待解析 1解析中 2已完成 3失败',
  status             TINYINT  NOT NULL DEFAULT 0 COMMENT '审核状态 0待审核 1审核中 2已完成 3失败',
  total_risk_count   INT      NOT NULL DEFAULT 0,
  major_risk_count   INT      NOT NULL DEFAULT 0,
  general_risk_count INT      NOT NULL DEFAULT 0,
  started_at         DATETIME DEFAULT NULL,
  finished_at        DATETIME DEFAULT NULL,
  error_msg          TEXT     COMMENT '失败原因',
  created_at         DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at         DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_file (contract_file_id),
  KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

文件路径约定由 `task_id` 推导，后端代码统一通过 `task_id` 拼接路径，无需在表中存冗余字段。

#### 审核结果条目表 `audit_result_item`

```sql
CREATE TABLE audit_result_item (
  id                BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  task_id           BIGINT UNSIGNED NOT NULL,
  category_id       BIGINT UNSIGNED NOT NULL COMMENT '规则分类ID（审核维度）',
  category_name     VARCHAR(100)    NOT NULL COMMENT '维度名称冗余，防止改名影响历史',
  title             VARCHAR(500)    NOT NULL COMMENT '风险标题',
  risk_level        TINYINT         NOT NULL COMMENT '1重大风险 2一般风险',
  risk_summary      TEXT            NOT NULL COMMENT '风险概述',
  modify_suggestion JSON            NOT NULL COMMENT '修改方案，字符串数组',
  clause_name       VARCHAR(200)    COMMENT '所属条款名称',
  hit_text          TEXT            COMMENT '命中的合同原文片段',
  hit_block_no      INT             COMMENT '命中的块序号（对应 blocks.json 的 no）',
  hit_page          INT             COMMENT '命中页码',
  hit_bbox_x        FLOAT           COMMENT 'PDF user space x（points）',
  hit_bbox_y        FLOAT           COMMENT 'PDF user space y（points）',
  hit_bbox_w        FLOAT           COMMENT 'bbox 宽度',
  hit_bbox_h        FLOAT           COMMENT 'bbox 高度',
  sort_order        INT             NOT NULL DEFAULT 0,
  created_at        DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_task (task_id),
  KEY idx_task_category (task_id, category_id),
  KEY idx_task_level (task_id, risk_level)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 审核维度统计表 `audit_dimension_stat`

```sql
CREATE TABLE audit_dimension_stat (
  id            BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  task_id       BIGINT UNSIGNED NOT NULL,
  category_id   BIGINT UNSIGNED NOT NULL,
  category_name VARCHAR(100)    NOT NULL,
  total_count   INT             NOT NULL DEFAULT 0,
  major_count   INT             NOT NULL DEFAULT 0,
  general_count INT             NOT NULL DEFAULT 0,
  PRIMARY KEY (id),
  UNIQUE KEY uk_task_category (task_id, category_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 5. 核心流程

```
1. 用户上传合同文件
   → 写 contract_file，创建 audit_task（parse_status=1）
   → 存原始文件：contracts/task_{task_id}/original.{ext}

2. 异步格式转换 + docling 解析
   .doc  → LibreOffice → .docx → PDF
   .docx → LibreOffice → PDF
   .pdf  → 直接使用
   → 存 converted.pdf：contracts/task_{task_id}/converted.pdf
   → docling 解析 PDF → 按行输出 → 存 lines.json：contracts/task_{task_id}/lines.json
   → 更新 audit_task.parse_status=2

3. 触发审核（parse_status=2 后自动）
   → 更新 audit_task.status=1
   → 从配置中心读取启用的 rule_category + risk_point

4. 按维度分批调用大模型
   → 对每个 rule_category，读 blocks.json 构造 Prompt（只传 no + text，不传坐标）
   → 大模型返回该维度风险 JSON 数组（含 hit_block_no）
   → 后端根据 hit_block_no 从 blocks.json 回查 page + bbox，写入 audit_result_item
   → 更新 audit_dimension_stat
   → 所有维度完成 → 更新 audit_task.status=2，汇总风险计数

5. 前端轮询 status=2，进入详情页
```

**大模型单条输出格式：**

```json
{
  "title": "主体信息缺失风险",
  "risk_level": 1,
  "risk_summary": "甲方信息不完整，名称、地址、法定代表人缺失...",
  "modify_suggestion": [
    "补全甲方名称、地址、法定代表人及统一社会信用代码",
    "明确责任主体、履行条件和确认方式"
  ],
  "clause_name": "主体信息条款",
  "hit_text": "甲方：华城数字建设有限公司",
  "hit_block_no": 3
}
```

`category_id/name` 由后端按当前批次 rule_category 填入，大模型无需感知。

## 6. API 设计

> 基础路径：`/api/v1/audit`

### 6.1 上传合同文件

```
POST /api/v1/audit/upload
Content-Type: multipart/form-data
```

```json
// 响应
{ "code": 0, "data": { "file_id": 1, "task_id": 123 } }
```

上传后自动创建任务，触发异步转换 + 解析 + 审核流程。

```
GET /api/v1/audit/tasks/:task_id/status
```

## 8. 关键设计说明

**以 task_id 为文件目录**：文件路径约定为 `contracts/task_{task_id}/`，后端通过 task_id 拼接，无需在表中存冗余路径字段，一个任务的所有文件（原始/PDF/解析结果）集中存放，便于清理。

**格式转换链路**：.doc 先经 LibreOffice 转 docx 再转 PDF；.docx 直接 LibreOffice 转 PDF；.pdf 直接使用。最终统一交给 docling 解析生成 `blocks.json`。

**`blocks.json` 双用途**：① 构造大模型 Prompt 时只传 `no + text`（不含坐标），大模型返回 `hit_block_no`；② 前端定位时后端根据 `hit_block_no` 从 `blocks.json` 回查 `page + bbox`，写入 `audit_result_item` 供前端 pdf.js 高亮渲染。坐标为 PDF user space（points，原点左下角），与 pdf.js viewport 坐标系一致，无需转换。

**`parse_status` 与 `status` 分离**：文件解析和审核是两个异步阶段，分别用独立字段跟踪，前端轮询可区分「解析中」和「审核中」两种进度状态。

**维度统计预聚合**：每批调用完成后即时写 `audit_dimension_stat`，Tab 计数直接读此表，无实时 COUNT。
