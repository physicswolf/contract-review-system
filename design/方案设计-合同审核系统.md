### 1.2 核心业务流程

```text
登录
  → 合同审核
      步骤1 上传合同（拖拽 / 选择 / 演示）→ 上传+解析进度条
      步骤2 确认审核设置
            · AI 识别合同类型（置信度，可手动改）
            · 选择审核立场（甲方 / 乙方）
            · 勾选审核点（默认推荐，可增减，跨合同类型自由组合）
            · 自定义审核目标
      步骤3 查看审核结果
            · 左栏：Word 原文（风险条款高亮）
            · 右栏：按审核维度分组 + 高/低/无风险筛选
            · 每条：风险概述 / 风险分析 / 修改方案 / 所属条款
            · 操作：定位原文、忽略本条、导出审核意见（docx）
应用中心：检索历史合同 → 查看审核详情
配置中心：审核点、合同类型、风险分级规则的后台维护
```

### 1.3 原型透露的关键设计决策

1. **合同类型与审核点解耦**：合同类型只用于识别、归档、检索和「推荐」审核点；实际审核规则由本次勾选的审核点决定。
2. **立场驱动**：甲方/乙方立场会影响分析侧重与修改建议。
3. **风险分级按「风险类型」定义**：不再用全局高中低，而是每个风险类型（权利义务失衡 / 履约保障不足 / 合规与权属风险）各自限定高/低/无风险标准。
4. **审核点 = 可配置的检索规则**：每个审核点含「原文示例 + 风险点 + 风险分析 + 修改示例」，这正是**条款召回 + 风险研判**的知识库。

> 这套设计本质上是一个「可配置审核点库 + 合同条款召回 + 立场化 LLM 研判 + 结构化报告导出」系统，与你已有的 LiteParse 解析 + LLM 分析 demo 高度衔接。

## 二、四大功能选型分析

### 2.1 Agent 框架选型

#### 候选对比


| 框架                             | 定位              | 优点                                              | 缺点                       | 适配度   |
| ------------------------------ | --------------- | ----------------------------------------------- | ------------------------ | ----- |
| **LlamaIndex（Workflow + RAG）** | 数据/RAG/Agent 一体 | 已用 LiteParse，同生态；RAG 检索能力强；Workflow 事件驱动适合固定流水线 | 多 Agent 能力非强项            | ★★★★★ |
| LangGraph                      | 图状态机编排          | 复杂分支/状态控制强、生态大                                  | 偏重、需引入整套抽象；RAG 要另配       | ★★★☆  |
| Dify                           | 低代码平台           | 出原型快、自带界面                                       | 私有化部署与自定义模型集成受限、可控性差     | ★★☆   |
| CrewAI / AutoGen               | 多 Agent 协作      | 角色分工、自治协作                                       | 你的流程是固定流水线，不需要自治调度，反增复杂度 | ★★    |
| 纯手写（OpenAI SDK）                | 自己编排            | 最轻、最可控                                          | 召回/索引/重排要自己造轮子           | ★★★   |


#### 结论与推荐

- 合同审核是**固定流水线**（上传→解析→分类→召回→研判→汇总→导出），**不需要多 Agent 自治**。
- 你的真正核心是**条款召回（RAG）**，而 LlamaIndex 同时覆盖**编排（Workflow）与检索（RAG）**，且与已用的 LiteParse 同生态。
- **推荐：LlamaIndex Workflow 作为编排层 + LlamaIndex RAG 作为召回层**。
- 暂不引入多 Agent；保留「纯手写 pipeline」作为 demo 阶段的轻量过渡，逐步替换为 Workflow。

```text
LlamaIndex Workflow（编排）
  ├─ Step 解析（docx→python-docx / pdf→docling，pymupdf4llm·MinerU 兜底）
  ├─ Step 合同分类
  ├─ Step 章节切分 + 建索引
  ├─ Step 按审核点并行召回 + LLM 研判（立场化）
  └─ Step 汇总报告 + 导出
```

### 2.2 合同分类设计

原型要求：上传后 **AI 自动识别合同类型并给置信度**，用户可改；合同类型配有「定义说明 + 识别关键词示例 + 推荐模板」。

#### 候选方案


| 方案                        | 原理                                | 优点             | 缺点                 |
| ------------------------- | --------------------------------- | -------------- | ------------------ |
| **LLM few-shot 分类（推荐主路）** | 将候选类型 + 定义 + 关键词输入生成模型，输出类型 + 置信度 | 零训练、可解释、易扩展新类型 | 依赖提示词质量            |
| 关键词规则（推荐做前置/兜底）           | 命中「采购/供货/价款」等关键词                  | 快、可解释          | 脆弱，易漏判             |
| 向量相似度                     | 合同 embedding 与类型中心向量比对            | 无需 LLM、快       | 需 embedding，边界类型易混 |
| 微调分类器                     | 训练专用模型                            | 精度高            | 需标注数据，demo 阶段过重    |


#### 推荐：关键词预筛 + LLM few-shot 主判 + 置信度

```text
合同首段/关键段
  → 关键词规则快速预测候选类型（缩小范围、加速）
  → LLM few-shot：给定类型定义+关键词，输出 {类型, 置信度, 依据}
  → 置信度 < 阈值 或 命中多类 → 标记「待人工确认」，前端高亮可改
```

- 与原型「类型编码 / 定义说明 / 识别关键词示例 / 推荐模板」字段**一一对应**，配置中心可直接维护。
- 合同类型仅用于**推荐审核点**，不强绑定，符合「解耦」设计。

### 2.3 合同解析及选型

> 本节结论已更新为**六框架横向对比（python-docx / pymupdf4llm / MinerU / LiteParse / docling / PyMuPDF）的本机实测结果**，详见 `Parser选型-六框架对比.md`。结论较早期「LiteParse 默认」方案有调整。

#### 六框架实测要点（真实合同样本，Apple Silicon 纯 CPU）

| 框架 | 支持格式 | 速度 | 结构化产出 | 本项目定位 |
| ---- | ---- | ---- | ---- | ---- |
| **python-docx** | docx | 极快(22ms) | 段落/原生样式/表格 | ✅ **Word 首选** |
| **docling** | pdf/docx等 | 中(实测0.37~0.40s/页,CPU) | Markdown/JSON(模型识别) | ✅ **PDF 默认（准确率优先，章节切分最佳）** |
| **pymupdf4llm** | pdf | 中(1.5~2.8s) | **Markdown**(标题+表格) | 🔶 高吞吐/轻量备选 |
| **PyMuPDF** | pdf | 极快(142ms) | 裸文本 | 🔶 高吞吐裸文本兜底 |
| **LiteParse** | pdf / docx* | PDF极快 / Word失败 | 文本+字号坐标 | ⚠️ Word 路径下线，PDF 降级兜底 |
| **MinerU** | pdf/图片 | 慢(CPU可跑/建议GPU) | Markdown/JSON | 🔶 扫描件/复杂版面专项兜底 |

\* LiteParse 的 docx 经 LibreOffice 转换，本次实测 **46KB docx 转换超时（>120s）失败**，不可用于生产。

#### 章节切分效果（本项目核心关注）★

同一份合同（真实结构「抬头 + 第一条～第十七条」）各框架识别出的章节标题对比：

| 框架 | 识别标题数 | 效果 | 说明 |
| ---- | ---- | ---- | ---- |
| **docling** | 24 | ✅ 最佳 | 第一～十七条全部正确，几乎无噪声（模型识别 section_header，不靠字号/正则） |
| **pymupdf4llm** | 36 | 🔶 中 | 条款全召回，但混入「甲方：/地址：」等正文碎片，需噪声过滤 |
| LiteParse+现有规则 | 131 | ⚠️ 过度切分 | 子条款(4.1.1)被当标题，粒度过碎 |
| PyMuPDF+字号规则 | 2 | ❌ 失效 | 合同标题与正文同字号(14pt)，字号方案不可靠 |
| python-docx | 28 | 🔶 依赖样式 | 样本无 Heading 样式，纯靠正则效果打折 |

> 关键发现：合同「第X条」标题常与正文**同字号**，**纯字号/纯正则切分在合同场景不可靠**；**docling 用模型识别标题，章节切分质量明显最优**。

#### 推荐：分格式 + 分难度的双层路由

```text
.docx → python-docx（原生样式/表格，快且稳）
.pdf  → docling（模型识别标题，准确率优先，默认）
         ├─ 高吞吐/轻量需求 → pymupdf4llm / PyMuPDF 备选
         └─ 扫描件 / 复杂版面 / 公式
                → MinerU（专项兜底，建议 GPU）
```

三句话结论：
1. **Word 用 python-docx**：替换 LiteParse 的 Word 路径，解决转换超时与不稳定，直接拿到标题样式 + 表格。
2. **PDF 默认 docling（准确率优先）**：用模型识别的标题做章节切分，质量最佳、阅读顺序正确；纯 CPU 即可（0.37s/页、整篇合同十几秒），首次需联网下模型，内网须预缓存权重。
3. **轻量/重型按需**：高吞吐或极轻部署场景用 pymupdf4llm / PyMuPDF；扫描件/复杂版面用 MinerU 定向兜底（建议 GPU）。

> 注：docling / MinerU **均可本机纯 CPU 运行**，GPU 仅用于提升吞吐与开启 OCR/复杂表格模型时的延迟，并非「能否运行」的前提。

### 2.4 合同条款召回（核心）

这是整个系统的技术核心，对应原型的「审核点 → 命中合同条款 → 风险研判 → 定位原文」。

#### 召回方向

采用**审核点驱动召回**（与原型一致）：

```text
for 每个用户勾选的审核点:
    query = 审核点描述 + 原文示例 + 风险点说明
    corpus = 当前合同的条款/章节切片
    → 检索 Top-K 相关条款
    → 立场化 LLM 研判（甲方/乙方）：是否命中风险 + 等级 + 修改建议
    → 输出风险卡片（含原文定位 span）
```

#### 检索技术栈


| 组件        | 推荐                                 | 说明                     |
| --------- | ---------------------------------- | ---------------------- |
| 切分        | 章节切分 + 滑窗                          | 复用 `splitter.py`，条款级粒度 |
| Embedding | **Qwen3-Embedding-8B（GPU）**        | MTEB 多语言榜第一；与生成模型同家族   |
| 向量库       | Chroma（轻量）/ Milvus（规模化）            | demo 用 Chroma          |
| 混合检索      | BM25 + 向量                          | 合同有大量专有词，关键词召回不可少      |
| 重排        | **Qwen3-Reranker-4B**              | 2026 年开源精排最强档，显著降漏判    |
| 编排/接口     | LlamaIndex Retriever / QueryEngine | 与编排层统一                 |


> 模型部署细节见「第八节 模型选型（GPU 私有化）」。

## 三、整体架构方案

```text
┌─────────────────────────── 前端 Vue3 + Element Plus ───────────────────────────┐
│ 登录 │ 合同审核(3步向导) │ 应用中心 │ 审核结果(双栏) │ 配置中心 │ 个人设置        │
└───────────────────────────────────────┬───────────────────────────────────────┘
                                         │ REST / SSE（流式进度）
┌───────────────────────────── 后端 FastAPI ────────────────────────────────────┐
│  LlamaIndex Workflow 编排                                                       │
│   1 解析   parser: docx→python-docx / pdf→docling（pymupdf4llm·MinerU 兜底） │
│   2 分类   classifier: 关键词预筛 + LLM few-shot（置信度）                       │
│   3 切分   splitter: 章节切分（docling 模型标题为主，pymupdf4llm 备选）          │
│   4 索引   indexer: 混合检索（BM25 + Qwen3-Embedding）+ Qwen3-Reranker          │
│   5 召回   retriever: 审核点驱动 Top-K 条款                                      │
│   6 研判   analyzer: 立场化生成模型逐审核点研判 + 修改建议                       │
│   7 汇总   reporter: 风险分级聚合 + docx 报告导出                                │
└───────────────────────────────────────┬───────────────────────────────────────┘
        ┌────────────────────────────────┼────────────────────────────────┐
   Qwen3-32B 生成(GPU)         Qwen3-Embedding + Reranker(GPU)        向量库 Chroma
                                                                      关系库 SQLite/PG
```

### 配置中心数据模型（对齐原型）

```text
合同类型 ContractType         审核点 AuditPoint            风险类型 RiskType
  - 编码                        - 名称                       - 名称（如权利义务失衡）
  - 名称/定义说明               - 审核维度                   - 高风险限定
  - 识别关键词                  - 风险类型(关联)             - 低风险限定
  - 推荐模板                    - 风险点[]（原文/风险/建议） - 无风险限定
  - 推荐审核点[]（弱关联）      - 启停状态(滑动)

审核任务 AuditTask = 合同 + 立场 + 合同类型 + 选中审核点[] + 自定义目标[]
审核结果 AuditResult = [风险卡片(审核点, 等级, 概述, 分析, 修改方案, 原文span, 是否忽略)]
```

4. 复用现有 `Result.vue` 双栏结构，扩展为「维度 Tab + 风险卡片 + 操作」。

## 五、落地路线（分阶段）

```text
阶段一 闭环跑通（低风险，纯本地，无 GPU）
  · 解析：Word→python-docx / PDF→docling（替换 LiteParse；预缓存 docling 模型权重）
  · 切分：PDF 用 docling 模型识别标题（section_header）；pymupdf4llm 作高吞吐备选
  · 分类：关键词 + LLM few-shot
  · 召回：先用 BM25/关键词（不依赖 embedding）
  · 研判：现有 analyzer 改为「审核点驱动 + 立场化」
  · 前端：三步向导 + 双栏结果（重构原型）

阶段二 引入 RAG（GPU 私有化）
  · 接入 Qwen3-Embedding + Chroma + 混合检索 + Qwen3-Reranker
  · 召回质量量化（建审核点 golden set）

阶段三 编排与增强
  · 用 LlamaIndex Workflow 重构 pipeline（并行召回、流式、重试）
  · 解析质量校验 + 兜底路由（空文本/扫描件→MinerU；高吞吐→pymupdf4llm）
  · MinerU fallback 接入（扫描件/表格密集合同，GPU）

阶段四 工程化
  · 配置中心后台（审核点/类型/风险分级 CRUD）
  · 权限、审计日志、报告模板、规模化向量库（Milvus）
```

## 六、关键风险与待确认项


| 事项           | 说明                          | 建议                               |
| ------------ | --------------------------- | -------------------------------- |
| **GPU 资源协调** | 模型私有化需 GPU 主机（华为云已有 GPU 资源） | 申请 1 台 80G GPU 或昇腾 910B，全栈单机部署   |
| embedding 落地 | RAG 召回前置条件                  | 部署 Qwen3-Embedding（与生成模型同栈，见第八节） |
| 出网受限         | 模型权重下载受限                    | 模型一次性离线拷贝进主机（含 docling/MinerU 模型预缓存） |
| PDF 章节过度切分   | LiteParse 现有规则把子条款当标题(切131章) | PDF 改用 docling 模型识别标题（章节切分实测最准）；pymupdf4llm 备选需噪声过滤 |
| Word 解析不稳    | LiteParse 走 LibreOffice，46KB docx 超时失败 | Word 改用 python-docx（毫秒级、原生样式/表格） |
| pymupdf4llm 许可 | AGPL-3.0（含 PyMuPDF）            | 内部研究无碍；若对外分发再评估，可换 MIT 系 docling |
| 召回精度评估       | 缺标注真值                       | 建审核点 golden set，量化召回率/准确率        |
| docling/MinerU 部署重 | 模型与依赖大，首次需联网下模型          | 纯 CPU 可跑（GPU 提速），独立解析服务，不进轻量镜像，模型预缓存 |


---

## 七、一页结论

1. **Agent 框架**：选 **LlamaIndex**（Workflow 编排 + RAG 召回），不引入多 Agent；与已用 LiteParse 同生态，同时覆盖流程编排与条款检索两大核心需求。
2. **合同分类**：**关键词预筛 + LLM few-shot + 置信度**，类型与审核点解耦，仅用于推荐。
3. **解析选型**：**Word→python-docx / PDF→docling（默认，准确率优先）**，pymupdf4llm 作高吞吐/轻量备选、MinerU 作扫描件/复杂版面专项兜底（详见 `Parser选型-六框架对比.md`）。章节切分实测以 **docling（模型识别标题）质量最佳**，字号/纯正则方案在合同场景不可靠。
4. **条款召回**（核心）：**审核点驱动 + 混合检索（BM25 + Qwen3-Embedding）+ Qwen3-Reranker + 立场化 LLM 研判**。
5. **模型**：全栈 **Qwen3 家族**（生成 Qwen3-32B + 召回 Qwen3-Embedding-8B + Qwen3-Reranker-4B），GPU 私有化，vLLM 部署，详见第八节。
6. **前端**：沿用 Vue3 + Element Plus，按原型重构「三步向导 + 双栏结果 + 配置中心」，进度用 SSE，结果支持原文定位与 docx 导出。
7. **路线**：先跑通审核闭环 → GPU 上 RAG → Workflow 重构 → 工程化。

## 八、模型选型（GPU 私有化）

华为云已有 GPU 主机资源，模型全部**私有化离线部署**，不依赖外部 API。整条链路统一在 **Qwen3 家族**（中文最强、全 Apache 2.0、与 LlamaIndex 无缝）。

### 8.1 模型栈


| 用途               | 推荐模型                         | 许可         | 说明                              |
| ---------------- | ---------------------------- | ---------- | ------------------------------- |
| 生成（分类 + 立场化风险研判） | **Qwen3-32B**                | Apache 2.0 | 主力；复杂条款可叠加 DeepSeek-R1 蒸馏做「推理脑」 |
| 召回 Embedding     | **Qwen3-Embedding-8B**       | Apache 2.0 | MTEB 多语言榜第一（70.58）              |
| 召回精排 Reranker    | **Qwen3-Reranker-4B**        | Apache 2.0 | 2026 年开源精排最强档之一                 |
| 解析（PDF 默认）         | **docling**（IBM，MIT）         | Apache/MIT | 模型识别标题，章节切分质量最佳；纯 CPU 可跑（模型预缓存） |
| 解析（高吞吐备选）         | **pymupdf4llm**               | AGPL-3.0   | 快、轻、pip 即用；章节标题需噪声过滤         |
| 解析扫描件兜底（可选）       | **MinerU**（自带 VLM）/ Qwen3-VL | -          | 扫描件、表格密集、复杂版面（建议 GPU）         |

### 8.4 部署拓扑

```text
GPU 主机（A100 80G / 昇腾 910B）
  vLLM（OpenAI 兼容推理引擎；昇腾用 vllm-ascend / MindIE）
    ├─ 服务A  Qwen3-32B           → /v1/chat/completions
    ├─ 服务B  Qwen3-Embedding-8B  → /v1/embeddings
    └─ 服务C  Qwen3-Reranker-4B   → rerank 接口
  向量库  Chroma（demo）/ Milvus（规模化）
```

要点：

1. vLLM 暴露 **OpenAI 兼容接口**，现有 `analyzer.py` 仅改 `base_url` + `model` 即可对接，无需重写；
2. 模型权重**一次性离线拷贝**进 GPU 主机，规避出网限制；
3. 昇腾环境把 vLLM 换为 `vllm-ascend` 或 MindIE，接口形态一致，上层代码不变。

### 8.5 推荐落地组合

```text

向量：Chroma（demo）→ Milvus（规模化）
硬件：A100/A800 80G ×1 或 昇腾 910B
```
